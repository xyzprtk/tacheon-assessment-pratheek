Task 1 / Product Scoping

Marketing Performance Reporter

An internal tool that automatically collects marketing data and delivers a daily briefing so analysts can focus on decisions instead of copying numbers.

The Problem

Every morning, analysts log into Meta Ads Manager, Google Ads, Google Analytics, Shopify, and other platforms to answer one question: how is our marketing performing, and where should we focus?

This routine creates three problems:

  • Inconsistency: different people pull data differently and get different answers.
  • Lost time: it takes 30 to 60 minutes when it should take five.
  • Key person risk: when the usual analyst is away, nobody knows how to get the report.

The Solution

The Marketing Performance Reporter is a simple internal pipeline that runs while you sleep.

At 6:00 AM UTC, it pulls data from all connected platforms. By 9:00 AM local time, it sends a clean daily briefing to Slack or email. The briefing includes:

  • A 7-day comparison so you spot trends without day-of-week noise.
  • A breakdown of the top 3 channels by spend.
  • Trend indicators that show what is moving up or down.
  • A one-sentence summary of the biggest change.
  • Flags for anything that looks unusual.
  • A clear note on where the data came from.

How It Works

The system follows an ELT pattern: Extract, Load, Transform.

Ingestion: Python scripts call the Meta, Google Ads, Shopify, and GA4 APIs.

Warehouse: Raw data lands in BigQuery, then moves through staging and mart tables.

Transformation: SQL calculates standardized metrics like ROAS, CPA, CTR, and CVR.

Delivery: A Jinja2 template formats the briefing, which is sent via Slack or email.

The Data Model

The core table is marketing_metrics_daily. It holds one row per client, per channel, per day, with normalized metrics ready for analysis.

Key design choices:

  • Partitioned by date for fast queries.
  • Clustered by client and channel for easy filtering.
  • Raw tables keep original API responses for 90 days so you can audit anything.

User Flows

We mapped out how people actually use the tool.

Daily workflow: The analyst opens Slack, reads the briefing in 2 to 3 minutes, and decides whether to dig deeper. If something looks off, they click through to BigQuery to verify.

Verifying a metric: If a number looks strange, the analyst queries the raw table directly. No black boxes.

Onboarding a client: Adding a new client means editing a YAML file. No code changes, no engineering tickets.

Handling incomplete data: If an API is down, the briefing says so clearly. It never shows a zero and pretends everything is fine.

Scope

In scope for v1

  • Daily automated briefings
  • Standardized metric definitions
  • Multi-tenancy from day one
  • Graceful handling of missing data
  • BigQuery access for verification

Out of scope

  • Predictive modeling
  • Real-time streaming
  • Self-service report builder
  • Client-facing branded reports
  • Custom dashboard UI

These exclusions are deliberate. We want to earn trust with clean, consistent data before adding complexity.

Success Metrics

We will know this works if:

  • Analysts spend under 5 minutes on the briefing instead of 30 or more.
  • Every analyst gets the same numbers for the same time period.
  • 80 percent of the team opens the briefing in the first month.
  • Analysts trust the numbers enough to use them in client meetings.
